# 模块02 实践示例

## 示例 1：提示词结构对比（前后对比）

### 任务：写一段产品发布通知

**无结构版本**
```
帮我写一个新产品发布的通知
```
→ 输出：通用模板，可能不符合实际需求

**结构化版本**
```
【角色】你是一名B2B SaaS公司的产品营销经理
【任务】为我们的新功能"AI数据分析仪表板"撰写一封发布通知邮件
【背景】目标受众：现有企业客户（财务总监和运营总监），他们最关心效率提升和成本节省
【格式】邮件结构：问候语 + 功能亮点（3条，每条20字内）+ 操作指引 + 签名，总长200字内
【约束】不使用技术术语；不提竞争对手；语气：专业但友好
【输出】直接输出邮件正文，不需要前言
```
→ 输出：完全符合规范，可直接使用

## 示例 2：JSON 格式精确控制

**任务**：从用户反馈中提取结构化数据

```
从以下用户反馈中提取信息，严格按JSON格式输出，不输出任何其他内容：

用户反馈："我在北京使用这款应用已经半年了，整体满意。
最喜欢的是实时导航功能，非常准确。但是电池消耗太快了，
而且有时候APP会突然崩溃。希望能加一个夜间模式。"

严格按以下JSON schema输出：
{
  "user_location": "城市名或null",
  "usage_duration": "使用时长描述",
  "overall_sentiment": "positive/negative/mixed",
  "liked_features": ["功能1", "功能2"],
  "issues": ["问题1", "问题2"],
  "feature_requests": ["需求1"],
  "priority_issue": "最严重的问题"
}
```

**实际输出**：
```json
{
  "user_location": "北京",
  "usage_duration": "半年",
  "overall_sentiment": "mixed",
  "liked_features": ["实时导航功能准确"],
  "issues": ["电池消耗快", "APP偶发崩溃"],
  "feature_requests": ["夜间模式"],
  "priority_issue": "APP崩溃"
}
```

## 示例 3：格式约束的力量

**场景**：同一任务，不同格式要求产生不同输出

```
任务：介绍Python的优点

格式A（无要求）→ 输出：长段落，约300字
格式B（"用5条要点，每条不超过15字"）→ 输出：5条精炼要点
格式C（"用Markdown表格，列：优点/应用场景/对比语言"）→ 输出：结构化表格
格式D（"用JSON数组，每项包含name和description字段"）→ 输出：机器可读JSON
```

## 示例 4：约束词的精确控制

**场景**：控制AI不过度发散

```
提示词（无约束）：
分析这份商业计划书，给出你的看法。
→ 输出：可能长达2000字，涵盖所有角度

提示词（有约束）：
分析这份商业计划书。
约束：
- 只聚焦3个最关键的风险点
- 每个风险点不超过50字
- 不讨论机遇或优势，只讨论风险
- 用"风险1/2/3："格式开头
→ 输出：精准、聚焦、可操作
```

## 示例 5：负面指令 vs 正面指令效果对比

**场景**：要求输出不包含解释

```
负面指令版本（不稳定）：
"告诉我今天是周几，不要解释，不要废话，直接说"
→ 模型有时仍会加"根据您的问题..."等前缀

正面指令版本（更稳定）：
"只输出今天是星期几，格式：星期X，不输出任何其他内容"
→ 模型更可靠地只输出"星期一"等
```

**原则**：正面指令（"只输出X"）比负面指令（"不要输出Y"）更可靠。
