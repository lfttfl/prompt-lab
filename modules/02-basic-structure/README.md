# 模块 02：提示词基础结构设计

> 难度：⭐⭐ 初级 | 建议学习时间：第3-4周（约10小时）

## 学习目标

完成本模块后，你将能够：
- 用六要素框架（RTCFCE）设计结构化提示词
- 精确控制 LLM 的输出格式（JSON、表格、列表等）
- 用负面指令和约束词限制不想要的行为
- 区分好提示词与差提示词的核心差距

---

## 1. 为什么需要结构化？

**差的提示词**：
```
帮我写一篇关于AI的文章
```

存在的问题：没有目标受众、没有规定长度、没有格式要求、没有专业深度要求、没有风格设定。
结果：每次输出差异巨大，难以满足实际需求。

**结构化提示词的价值**：可预测性、一致性、可维护性。

---

## 2. 六要素框架（RTCFCE）

### R - Role（角色）
定义 AI 在这次对话中扮演的角色。

```
你是一位有10年经验的科技媒体记者，专注于AI领域报道，文风深入浅出。
```

**作用**：角色定义激活模型在训练数据中相关领域的知识和风格。

### T - Task（任务）
精确描述要完成的具体任务。

```
请为我写一篇关于大语言模型在医疗领域应用的科普文章。
```

**好任务描述**：使用动词开头（写/分析/总结/提取/转换）；包含具体结果描述；避免模糊词（"好一点"、"更好"）。

### C - Context（背景）
提供完成任务所需的背景信息。

```
目标读者：没有技术背景的医疗从业者（医生、护士）。
他们对AI持谨慎态度，关注实际应用价值和风险。
发布平台：医院内部刊物。
```

只提供真正有用的背景，不要把所有信息都堆进去。

### F - Format（格式）
明确规定输出的结构和形式。

```
格式要求：
- 总长度：800-1000字
- 结构：标题 + 引言(100字) + 3个应用案例(各200字) + 结语(100字)
- 每个案例需包含：具体场景、当前挑战、AI解决方案
- 使用小标题分隔各部分
```

### C - Constraints（约束）
规定 AI 不应该做什么。

```
约束条件：
- 不使用技术术语（如"参数"、"训练集"等）
- 不夸大AI能力，保持客观中立
- 不涉及具体产品或公司名称
- 每个句子不超过30字
```

### E - Examples（示例）
提供期望输出的参考示例（可选，但效果显著）。

```
引言风格参考（仅供参考，不是内容要求）：
"当医生需要在千份病历中寻找类似病例时，AI在5秒内完成了人工需要5天的工作..."
```

## 3. 完整示例：从差到好

### 差版本
```
写一篇AI文章给医生看
```

### 好版本
```
【角色】你是一位有10年经验的医疗科技记者，擅长向非技术读者解释前沿技术。

【任务】写一篇关于AI在医疗影像诊断中应用的科普文章。

【背景】目标读者：三级医院放射科医生，有丰富临床经验但对AI不熟悉。
核心关切：AI能否提高诊断准确率？会取代医生吗？需要多少培训？

【格式】
- 总长度：600-800字
- 结构：引言 → 3个应用案例 → 医生角色定位 → 未来展望
- 使用二级标题
- 每段不超过100字

【约束】
- 不使用"算法"、"模型"、"数据集"等技术词汇
- 必须强调AI是辅助工具，最终决策权在医生
- 不引用具体产品

【输出】直接开始写文章，不需要说明你的思路。
```

## 4. 输出格式控制

### JSON 格式输出
```
分析以下产品评论，严格以JSON格式输出，不要有任何其他文字：

评论：「这款耳机音质超棒，戴了3小时耳朵不疼，但充电速度太慢了」

{
  "sentiment": "positive/negative/mixed",
  "score": 1-5的整数,
  "pros": ["优点1"],
  "cons": ["缺点1"],
  "key_attributes": {
    "音质": "positive/negative/neutral",
    "舒适度": "positive/negative/neutral",
    "充电": "positive/negative/neutral"
  }
}
```

### Markdown 表格
```
对比以下3款工具，输出为Markdown表格，包含列：工具名、月费、核心功能、适合场景
[工具信息]
```

### 分步列表
```
将以下操作步骤整理为带编号的清单，每步一行，不超过20字：
[操作说明]
```

## 5. 负面指令使用

负面指令告诉模型**不要做什么**，对控制不想要的行为很有效。

| 场景 | 负面指令 |
|------|---------|
| 防止废话 | "不要在回答前解释你的思路" |
| 控制长度 | "不要超过200字" |
| 避免免责声明 | "不要添加任何免责声明" |
| 控制格式 | "不要使用Markdown，只输出纯文本" |

**注意**：负面指令不如正面指令稳定。优先用正面指令（"只输出X"比"不要输出Y"更可靠）。

---

## 6. 高频错误与修正

| 错误模式 | 问题提示词 | 修正版本 |
|---------|---------|--------|
| 指令模糊 | "改进这段文字" | "优化逻辑结构，使论点更清晰，不改变内容" |
| 格式缺失 | "列出5个优点" | "列出5个优点，每条用'✓'开头，不超过20字" |
| 上下文不足 | "回复这封邮件" | "以销售经理身份回复，重点回应客户的价格疑虑" |
| 约束冲突 | "写一篇深度的100字文章" | "深度"和"100字"冲突，需明确优先哪个 |

---

## 本模块总结

| 要素 | 核心问题 | 常见错误 |
|------|---------|---------|
| Role | AI是谁？ | 角色太模糊（"你是一个助手"） |
| Task | 要做什么？ | 动词不精确（"帮我处理"） |
| Context | 需要什么背景？ | 信息过多或过少 |
| Format | 输出什么样？ | 没有明确格式导致不可预测 |
| Constraints | 不能做什么？ | 只写正面指令，忽略边界控制 |
| Examples | 期望什么结果？ | 示例质量差，反而误导模型 |
