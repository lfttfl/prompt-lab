# 模块06 实践示例

## 示例 1：完整 A/B 测试案例

**任务**：分析客户邮件，判断客户情绪并推荐回复策略

**测试变量**：是否包含具体的情绪分类标准

**版本A（无标准）**：
```
分析以下客户邮件的情绪，并推荐回复策略。
[邮件内容]
```

**版本B（有标准）**：
```
分析以下客户邮件，按以下标准判断情绪等级：
- 愤怒：使用感叹号、强烈投诉词（"要投诉"、"要退款"、"太差了"）
- 失望：表达预期落差，语气平和但消极
- 中性：纯事实描述，无明显情绪
- 满意：积极词汇，表达认可

输出格式：
情绪等级：[愤怒/失望/中性/满意]
核心诉求：[一句话]
推荐策略：[具体行动，不超过3条]

[邮件内容]
```

**测试结果（10封邮件）**：
| 指标 | 版本A | 版本B |
|------|-------|-------|
| 情绪判断准确率 | 70% | 95% |
| 格式合规率 | 60% | 100% |
| 策略可操作性（人工评分1-5）| 2.8 | 4.3 |

结论：明确标准 + 格式要求，显著提升一致性和可用性。

## 示例 2：失败模式诊断过程

**问题现象**：提示词有时输出JSON，有时输出普通文字。

**诊断步骤**：

```
步骤1：收集失败案例
→ 发现失败多发生在用户输入较短（<10字）时

步骤2：隔离变量
→ 测试"输入长度"是否是触发因素
→ 验证：<10字的输入中，JSON格式合规率60%；>20字的输入，合规率95%

步骤3：假设根本原因
→ 假设：短输入时模型认为任务太简单，直接回答而跳过格式

步骤4：验证和修复
→ 修复：在格式要求后加"无论输入内容长短，都必须严格按JSON格式输出"
→ 验证：修复后短输入JSON合规率提升到92%
```

## 示例 3：版本迭代记录

**任务**：自动生成商品标题（电商）

```
v1.0（基础版）：
"为以下商品信息生成一个吸引人的标题"
→ 问题：标题长度不一致，有时太长超出平台限制

v1.1（加长度限制）：
"...生成标题，不超过30个字"
→ 问题：标题虽短，但有时缺少关键词（影响SEO）

v1.2（加关键词要求）：
"...必须包含品牌名和核心卖点关键词，不超过30字"
→ 问题：有时会忽略平台风格（某平台偏好感叹号风格）

v1.3（加平台风格）：
"...适合京东平台风格（简洁专业），必须包含品牌+核心卖点，不超过30字"
→ 结果：准确率从45%提升到82%

关键洞察：每个版本只改一处，才能准确定位每次改进的贡献。
```

## 示例 4：成本优化实验

**场景**：系统提示词从300 Token 压缩到 150 Token

```
原版系统提示（约300 Token）：
"你是一个非常有经验的专业助手，具有丰富的知识储备。请你在回答问题的时候，
保持专业的态度，同时使用清晰明了的语言，让用户能够理解。你需要根据用户的
需求来提供帮助，如果用户问到你不确定的问题，请告知用户你不确定。另外，
你的回答应该保持简洁，不要使用过多的废话和重复..."

压缩版（约80 Token）：
"你是专业助手。回答要准确、简洁、清晰。不确定时明确说明。"

测试结果（100条测试）：
- 输出质量：原版 4.2分 / 压缩版 4.0分（差异不显著）
- 每次调用节省：约220个输入Token（日均1万次调用节省220万Token/天）
```
