## 1. RAG 提示词设计

RAG（检索增强生成）将外部知识库与 LLM 结合。提示词贯穿整个流程。

### RAG 流程中的关键节点

```
用户问题 → [查询改写] → 知识库检索 → [上下文注入] → [引用格式] → 最终回答
```

### 1.1 查询改写提示词

```
你是搜索查询优化器。将用户问题改写为2-3个更精确的检索查询。

要求：
- 扩展同义词和相关概念
- 分解复合问题为独立子查询
- 使用专业术语（如原问题是口语）
- 每行一个查询，不编号

用户问题：{user_question}
改写后的查询：
```

### 1.2 上下文注入提示词

```
你是[应用名]的知识库助手，只基于以下参考资料回答问题。

【参考资料】
{retrieved_chunks}

【用户问题】
{user_question}

【回答规则】
1. 只使用参考资料中的信息，不添加外部知识
2. 参考资料中没有相关信息时，明确说明
3. 在回答末尾标注来源：[来源：文档名，第X段]
4. 回答长度：100-300字

【回答】
```

### 1.3 引用验证提示词

```
检查以下回答是否与参考资料一致，标出不准确或无依据的内容：

参考资料：{context}
AI回答：{answer}

输出：
- 准确：[内容]（来源：[段落]）
- 需核实：[内容]（原因：[说明]）
- 不准确：[内容]（与资料冲突：[说明]）
```

## 2. Function Calling 提示词

### 工具描述设计原则

```json
{
  "name": "search_products",
  "description": "在产品数据库中搜索商品。用户询问产品价格、库存、规格或需要推荐时调用。不适用于订单查询或售后问题。",
  "parameters": {
    "query": {
      "type": "string",
      "description": "搜索关键词，可以是产品名称、类别或特性"
    },
    "category": {
      "type": "string",
      "description": "产品类别：electronics/clothing/food，不确定则留空",
      "enum": ["electronics", "clothing", "food"]
    }
  },
  "required": ["query"]
}
```

**关键要点**：
- `description` 字段说明**何时调用**和**何时不调用**
- 参数描述要说明类型、范围
- 列出可选值（enum）比让模型自由填写更可靠

### 系统提示词中的工具使用指南

```
你有以下工具可用：[工具列表]

使用原则：
1. 只在需要外部数据时才调用工具
2. 单次回答中最多调用3个工具
3. 工具调用失败时告知用户并提供替代方案
4. 不确定工具参数时，向用户确认而非猜测
```

## 3. ReAct Agent 提示词

ReAct（Reasoning + Acting）是最广泛使用的 Agent 框架：

```
你是可以使用工具解决问题的AI助手。

可用工具：
- search(query)：搜索互联网信息
- calculator(expression)：计算数学表达式
- get_weather(city)：获取天气信息

解决问题时，循环执行直到有最终答案：

思考：[分析当前情况，决定下一步]
行动：[工具名称]([参数])
观察：[工具返回的结果]
思考：[根据结果决定下一步或得出答案]
...
最终答案：[给用户的回答]

限制：最多10个步骤；不确定时询问用户。
```

## 4. 提示词链（Prompt Chaining）

将复杂任务拆分为多个步骤：

```python
def analysis_pipeline(document):
    # 步骤1：摘要
    summary = llm(f"用200字总结核心内容：\n{document}")
    
    # 步骤2：关键点提取
    key_points = llm(f"提取5个最重要的论点，JSON数组输出：\n{summary}")
    
    # 步骤3：批判性分析
    analysis = llm(f"批判性分析以下论点，指出逻辑漏洞：\n{key_points}")
    
    # 步骤4：生成报告
    report = llm(f"整理成结构化专业报告：\n{analysis}")
    return report
```
