# 模块 01：基础原理与核心概念

> 难度：⭐ 入门 | 建议学习时间：第1-2周（约10小时）

## 学习目标

完成本模块后，你将能够：
- 用自己的话解释 LLM 的工作原理
- 理解 Token 化机制及其对提示词的影响
- 知道上下文窗口的限制，并据此设计提示词
- 掌握 temperature、top-p 等关键参数的调节方法
- 理解系统提示（System Prompt）与用户提示的区别

---

## 1. 大语言模型（LLM）是什么？

大语言模型是在海量文本上训练的神经网络，本质是一个**下一个词预测机器**。给定已有文本，LLM 计算词表中每个词的概率，然后根据策略选取下一个词。

```
输入：  "今天天气"
预测：  "很好"(45%) / "不错"(22%) / "真棒"(15%) / ...
输出：  "很好"（采样选择）
```

**这意味着什么？**

- LLM **没有真正的"理解"**，它在做统计模式匹配
- 提示词的措辞会深刻影响结果，因为它改变了上下文分布
- 相同提示词每次运行可能产生不同结果（除非 temperature=0）
- LLM 不"知道"正确答案，它在预测"合理的下一个词"

## 2. Tokenization（词元化）

LLM 不直接处理字符或词语，而是处理 **Token（词元）**。

### 什么是 Token？

Token 是模型实际处理的文本单元：
- 英文：约 1 个单词 ≈ 1.3 个 Token
- 中文：约 1 个汉字 ≈ 1-2 个 Token
- 标点符号、空格通常独立成 Token
- 不常见词会被拆分成多个 Token

### Token 为什么重要？

1. **成本计算**：API 按 Token 计费
2. **上下文限制**：模型有最大 Token 数限制
3. **理解偏差**："GPT-4" 可能被切分为 ["GP", "T", "-", "4"]，影响专有名词的理解
4. **数字处理**：大数字往往被逐位 Token 化，影响数学能力

**实用建议**：避免在关键词中间断行；专有名词用常见写法；中英文混用时 Token 消耗会增加。

## 3. 上下文窗口（Context Window）

上下文窗口是 LLM **在一次推理中能"看到"的最大 Token 数**。

| 模型 | 上下文窗口 |
|------|---------|
| GPT-4o | 128K tokens |
| Claude Sonnet 4.6 | 200K tokens |
| Gemini 2.0 Pro | 1M tokens |

### 上下文窗口的内容结构

```
┌─────────────────────────────────┐
│  System Prompt（系统提示）        │  ← 核心指令，优先级最高
│  ─────────────────────────────  │
│  对话历史（之前的轮次）            │  ← 模型的"短期记忆"
│  ─────────────────────────────  │
│  User Message（当前输入）         │  ← 用户当前问题
│  ─────────────────────────────  │
│  [模型生成区域]                   │  ← 模型输出
└─────────────────────────────────┘
```

### 关键认知

- **先出现的内容影响更大**：系统提示在最前面，有更强的锚定效应
- **"迷失在中间"效应（Lost in the Middle）**：模型对上下文中间位置的信息记忆较差
- **超出窗口 = 遗忘**：超过最大 Token 数的内容模型完全看不到
- **长上下文 ≠ 更好理解**：窗口越长，某些任务的注意力会越分散

**实用建议**：重要指令放在开头和结尾；精简系统提示，删除冗余内容。

## 4. 关键参数详解

### Temperature（温度）

控制输出的**随机性**，范围通常是 0.0 ~ 2.0：

| Temperature | 行为 | 适合场景 |
|-------------|------|--------|
| 0.0 | 确定性输出（总是选概率最高的词） | 代码生成、数据提取、事实问答 |
| 0.3-0.7 | 平衡确定性与创造性 | 文案写作、报告生成 |
| 0.8-1.2 | 高创造性，输出多样 | 头脑风暴、创意写作 |
| >1.5 | 输出可能不连贯 | 通常不推荐 |

### Top-p（核采样）

选取概率总和达到 p 的最小词集，在其中随机采样。

**建议**：一般只调 temperature 或 top-p 其中一个，不要同时修改。

### Max Tokens

限制模型输出的最大 Token 数。根据任务预估输出长度，再留 20% 余量。

## 5. 提示词的角色结构

所有提示词由以下角色构成：

### System Prompt（系统提示）
- 定义模型的角色、行为约束、输出格式
- 用户通常看不到，由开发者设置
- 优先级最高，贯穿整个对话

### Human Turn（用户输入）
- 用户实际发送的内容
- 包含具体任务描述和相关信息

### Assistant Turn（模型输出）
- 模型的回复
- 在多轮对话中，之前的回复会进入上下文

```
System: "你是专业的技术文档编辑，只使用中文，风格简洁专业。"
Human:  "帮我总结这篇论文：[论文内容]"
Assistant: [模型回复]
Human:  "将第一个观点展开成300字的详细说明"
Assistant: [模型基于完整上下文再次回复]
```

## 7. 常见误区

❌ **误区1：越长的提示词越好**
→ 精炼的提示词通常优于冗长的提示词。无关信息会稀释重要指令。

❌ **误区2：LLM 有"记忆"**
→ 每次调用都是独立的。"记忆"是通过将历史对话放入上下文实现的。

❌ **误区3：LLM 会"理解"意图**
→ LLM 做的是文本模式匹配，必须明确表达意图，不能依赖隐含理解。

❌ **误区4：高温度 = 更聪明**
→ 高温度增加随机性，不等于增加智能。

| 概念 | 核心要点 |
|------|--------|
| LLM工作原理 | 下一个词预测；统计模式匹配；不等于真正理解 |
| Token | 文本的基本处理单元；影响成本、限制和准确性 |
| 上下文窗口 | 模型的"工作记忆"；重要内容放首尾 |
| Temperature | 控制随机性；事实任务用低温，创意任务用高温 |
