# 模块03 实践示例

## 示例 1：Zero-shot vs Few-shot 准确率对比

**任务**：分类客服工单优先级（紧急/普通/低优先级）

**Zero-shot版本**
```
将以下客服工单分类为：紧急/普通/低优先级

工单："我的账户无法登录，影响今天的重要演示"
分类：
```
结果：有时分为"紧急"，有时分为"普通"，不稳定。

**Few-shot版本（3个示例）**
```
客服工单优先级分类规则示例：

工单："系统完全崩溃，所有用户无法访问" → 紧急（影响所有用户/业务停摆）
工单："我想修改一下我的个人资料照片" → 低优先级（纯外观/非功能性）
工单："订单支付失败，能帮我处理一下吗" → 普通（影响单用户核心功能）

现在分类：
工单："我的账户无法登录，影响今天的重要演示"
分类：
```
结果：稳定输出"紧急"，因为示例清晰地定义了"紧急"的判断标准。

## 示例 2：示例质量对结果的影响

**场景**：提取合同中的关键日期

**低质量示例（标注不一致）**
```
示例1：合同内容："甲方于2024年3月1日交付产品" → 日期：3月1日
示例2：合同内容："付款截止日期为2024年12月31日" → 日期：2024/12/31
```
问题：两个示例的日期格式不一致（"3月1日" vs "2024/12/31"），模型输出格式会不稳定。

**高质量示例（格式一致）**
```
示例1：合同内容："甲方于2024年3月1日交付产品" → 日期：2024-03-01
示例2：合同内容："付款截止日期为2024年12月31日" → 日期：2024-12-31
示例3：合同内容："合同有效期自签署之日起两年" → 日期：无明确日期
```
结果：稳定输出 YYYY-MM-DD 格式，且能正确处理无明确日期的情况。

## 示例 3：示例顺序的近因效应

**实验**：相同示例，不同排列顺序，影响输出倾向

```
顺序A（最后一个示例是正面）：
示例1: "配送太慢" → 负面
示例2: "价格公道" → 正面  ← 最后
→ 模型在模糊情况下倾向于输出"正面"

顺序B（最后一个示例是负面）：
示例1: "价格公道" → 正面
示例2: "配送太慢" → 负面  ← 最后
→ 模型在模糊情况下倾向于输出"负面"
```

**建议**：多样化排列示例，或将最典型的示例放最后。

## 示例 4：Few-shot 用于风格迁移

**任务**：将正式报告语言改写为微博风格

```
示例1：
原文："本季度营收同比增长23%，主要驱动因素为海外市场拓展。"
微博版："季报出来了！海外市场猛发力，营收涨了23%！"

示例2：
原文："产品研发投入同比增加15%，持续加强核心技术壁垒。"
微博版："研发钱没少花！技术护城河越来越深，好事儿。"

现在改写：
原文："公司完成新一轮融资，金额达5亿元人民币，投后估值80亿。"
微博版：
```

结果：模型准确捕捉到微博风格：口语化、有感叹、有emoji（如果示例有的话）。
