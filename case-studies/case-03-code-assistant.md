# 案例03：代码审查助手

## 业务背景

一家20人的初创团队，代码审查完全依赖人工，平均每个PR等待审查时间超过8小时。团队希望用AI做初步审查，让人工审查专注于架构和业务逻辑，而非基础问题。

## 需求分析

**功能需求**：
- 分析代码质量：可读性、命名规范、代码复用
- 识别潜在 bug：空指针、边界条件、异常处理
- 安全漏洞扫描：SQL注入、XSS、敏感信息泄露
- 性能建议：算法复杂度、数据库查询优化

**输出要求**：
- 严重性分级（Critical/Warning/Suggestion）
- 指出具体行号
- 提供修复建议（但不直接修改代码）
- 中文输出

---

## 设计过程

### v1.0 —— 问题：反馈过于模糊

```
你是代码审查专家，请审查以下代码并给出建议：
{code}
```

**问题**：输出类似"变量命名可以更清晰"，没有具体指出哪里、为什么、如何改进。

---

### v1.1 —— 加入审查维度，但结构仍混乱

```
请从以下4个维度审查代码：
1. 代码质量  2. 潜在bug  3. 安全性  4. 性能

{code}
```

**问题**：四个维度的问题混在一起，没有严重性分级，难以优先处理。

---

### v1.2 —— 加入 CoT + 结构化输出

```
你是一位有10年经验的高级工程师，专注于代码质量和安全审查。

请按以下步骤审查代码：

【第一步：快速扫描】
先快速浏览代码，识别编程语言、主要功能、代码规模。

【第二步：逐项分析】
按以下4个维度，逐行分析代码问题：
- 安全漏洞（优先级最高）
- 潜在Bug（空指针、边界条件、异常处理）
- 代码质量（可读性、命名、重复代码）
- 性能问题（算法复杂度、不必要的操作）

【第三步：输出报告】
严格按以下格式输出，不输出其他内容：

## 代码审查报告

**代码概述**：[一句话描述代码功能]

### 🔴 Critical（必须修复）
- [行X] 问题描述 | 修复建议：...

### 🟡 Warning（建议修复）
- [行X] 问题描述 | 修复建议：...

### 🔵 Suggestion（可选优化）
- [行X] 问题描述 | 修复建议：...

**总结**：Critical [N]个，Warning [N]个，Suggestion [N]个

---
【待审查代码】
语言：{language}
```{language}
{code}
```
```

---

### v1.3（最终版）—— 加入Few-shot示例 + 边界处理

在 v1.2 基础上增加：

```
【严重性定义】
- Critical：可能导致安全漏洞、数据丢失、系统崩溃的问题
- Warning：可能导致逻辑错误或难以维护的问题
- Suggestion：改善代码质量但不影响功能的建议

【示例输出片段】
### 🔴 Critical
- [行15] SQL拼接存在注入风险：`query = "SELECT * FROM users WHERE id=" + user_id`
  修复建议：使用参数化查询 `cursor.execute("SELECT * FROM users WHERE id=?", (user_id,))`

【特殊情况处理】
- 如果代码少于5行：输出"代码量过少，建议提供更完整的上下文"
- 如果代码超过200行：重点审查前50行，其余标注"[建议分批审查]"
- 如果发现严重安全漏洞：在报告最前面加上 ⚠️ 紧急安全警告
```

---

## 测试验证

**测试集**：50段不同语言的代码（Python/JavaScript/Java），已知问题清单作为标准答案

| 指标 | v1.0 | v1.3 |
|------|------|------|
| Critical问题检出率 | 45% | 88% |
| 误报率（误识别正常代码为问题）| 35% | 12% |
| 格式合规率 | 30% | 97% |
| 开发者满意度（1-5分）| 2.3 | 4.4 |

---

## 关键设计洞察

1. **CoT分步骤是关键**：先"扫描"再"分析"再"输出"，比直接要求输出报告准确率高40%
2. **严重性定义要量化**：不要说"重要问题"，要说"可能导致安全漏洞的问题"
3. **示例对格式稳定性至关重要**：一个示例输出可以使格式合规率从30%提升到97%
4. **边界情况要预先处理**：空代码、超长代码、非代码输入都要在提示词中说明
